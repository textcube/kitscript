# BuntOcean — Cinematic Water World

Three.js(r128) + Tween.js로 만든 단일 HTML 파일 기반의 3D 쇼케이스 페이지다. [city](../city)(BuntGames Alley Explorer)와 같은 아키텍처의 밤바다 버전 — 요트를 따라가는 3인칭 카메라로 z축 일렬 노드(부표) 사이를 항해하고, 해상 요새 빌보드를 더블클릭하면 프로젝트 소개 모달(유튜브/이미지 + 링크)이 뜬다.

* https://github.com/textcube/kitscript/

## 현재 구현 개요

- **월드 데이터**: `WORLD_DATA.nodes` 5개(Outer Harbor → Deep Abyss)가 z축 일렬(40 → -60)로 늘어서 있고, `navigateTo()`가 인덱스 순서로 순차 항해한다(city처럼 BFS 그래프는 아니고 단순 인덱스 순회). `WORLD_DATA.forts` 4개가 x=-25 한쪽에 빌보드 요새로 배치된다.
- **플레이어**: 요트(`createDetailedYacht`) — 헐/브릿지/레이더/네온 트림 + idle 부유 애니메이션(롤/피치/상하) + 웨이크 링.
- **바다(`createOcean`)**: 요트를 따라다니는 소형 정밀 평면(600×600, 64×64 세그먼트, ~4,200 정점)만 매 프레임 CPU에서 사인파로 변형하고, 20000×20000 대형 정적 평면이 원경/레이캐스트(`userData.isSea`)를 담당한다(A1 완료 — 아래 참고). 반투명 sheen 평면이 위에 겹쳐 있다.
- **환경**: 별 1600개(Points), 달 + 헤일로, 원경 고철 환초 20개(각각 부유 애니메이터), `Fog(0x0a2130, 120, 4200)`.
- **조명**: DirectionalLight(달) + AmbientLight + 수평선 PointLight 1개. 부표 PointLight 5개는 제거했다(A2 완료) — 동적 라이트 6→1개.
- **인터랙션**: `pointerdown`(단일클릭: 노드/바다)과 `dblclick`(빌보드 포커스)을 분리했다(city 패턴, B9 완료) — 바다 클릭=앞/뒤 노드 이동, 부표 클릭=해당 노드로, 빌보드 더블클릭=카메라 포커스+모달. 휠=FOV 줌(15~100), W/S·방향키 이동. 모달은 X 버튼/Escape/배경 클릭으로 닫힌다.
- **HUD**: CSS Grid(brand/actions/location named area), `display: contents` 미사용, `.brand-name`/`.hud-actions`/`.location-info`가 `#hud`의 직계 자식.

## 개선 이력 — city에서 검증된 패턴 이식 (완료)

[city/README.md](../city/README.md)에 기록된 개선 과정에서 검증된 패턴을 water에 이식했다. A(성능)와 B(UI/UX) 전 항목을 완료했다.

### A. 성능 (city 대응: 렌더링/성능 항목)
- ✅ **바다 정점 변형 부하 축소** — `createOcean()`을 이중 평면으로 분리했다. (a) 600×600/64×64 세그먼트의 소형 디테일 평면이 요트를 따라 `position.x/z`를 매 프레임 스냅하고, 사인파 계산에는 로컬 정점 좌표가 아니라 로컬좌표+평면 world position(월드 좌표)을 넣어 평면이 움직여도 파도가 "헤엄치듯" 밀리지 않게 했다. (b) 20000×20000/1×1 대형 정적 평면이 원경 색상과 바다 레이캐스트(`userData.isSea`)를 전담한다 — 평평하고 y=0이라 클릭 이동 판정에 그대로 쓸 수 있다. 디테일 평면은 `interactiveObjects`에서 제외했다. 정점 수 ~16,600 → ~4,200(디테일 평면만 변형)으로 줄었고, `renderer.info.render.triangles`가 14,632로 낮아진 것을 `preview_eval`로 확인했다(구 버전 128×128 평면 하나만으로도 32,768 트라이앵글이었던 것과 비교).
- ✅ **부표 PointLight 5개 제거** — `createMarkerBuoys()`에서 `PointLight`와 그 강도 펄스 애니메이션을 삭제했다. bulb(MeshBasic) opacity 펄스 + halo는 그대로 유지해 발광 느낌은 남는다. `scene.traverse`로 `isPointLight` 개수를 세어 1개(horizonGlow)만 남은 것을 확인했다.
- ✅ **pixelRatio 상한 2 → 1.5** — `renderer.setPixelRatio(Math.min(window.devicePixelRatio || 1, 1.5))`.
- ✅ **불필요한 유틸리티 정리** — `boot()`에서 `.guide-box span`을 JS로 덮어쓰던 블록을 제거하고, 최종 문구("🚢 CLICK SEA TO SAIL" 등)를 HTML에 직접 박아뒀다. 터치 기기에서는 `applyTouchAwareGuideText()`가 여전히 한국어 문구로 교체한다.

### B. UI/UX (city UI/UX 보강 6종 + 후속 수정 이식)
- ✅ **HUD를 CSS Grid로** — city와 동일한 named-area 그리드(`brand`/`actions`/`location`). `.brand-name`/`.hud-actions`/`.location-info`를 `#hud` 직계 자식으로 두고 `.brand-zone` 이중 사용 구조를 제거했다. `display: contents` 미사용.
- ✅ **모달 ESC/배경 클릭 닫기** — `onKey()` 최상단에서 `isMoving||isInspecting` 가드보다 먼저 Escape를 처리한다. 배경 클릭은 `onModalBackdropClick`에서 `e.target === e.currentTarget`로 판정.
- ✅ **노드 트래커** — `#node-tracker`/`buildNodeTracker()`/`updateNodeTrackerActive()`를 그대로 이식. 우측 세로 점 5개, 활성 노드 확대+글로우, 클릭 시 `navigateTo()`, 데스크톱 hover 라벨(폭 520px 이하 세로 화면에서는 숨김).
- ✅ **안내 pill 자동 숨김 + 중앙 정렬** — `scheduleGuideHide()`(9초) / `onFirstModalOpen()`(첫 모달) / `resetGuideIdleTimer()`(45초 유휴 재등장). `.guide-box`의 기본/`.revealed`/`.hidden-by-timer` 세 상태 모두 `translateX(-50%)`를 transform에 포함시켰다. `preview_eval`로 데스크톱과 390×844 양쪽에서 뷰포트 가로 중앙 오차 0px 확인.
- ✅ **로딩 페이드아웃 + HUD 스태거 등장** — `#loading.fade-out`(0.6초 opacity 트랜지션) 후 `transitionend`(700ms 폴백)에서 `display:none`. `boot()`에서 기존 동기 처리 대신 800ms `setTimeout`으로 지연시켜 실제로 페이드가 보이게 했다. `#hud`/`.guide-box`가 `.revealed` 클래스로 순차 페이드업.
- ✅ **빌보드 호버 피드백** — `createFortressBillboards()`에서 각 요새 네온 토러스에 `neonRef = { hoverBoost, mat, base }` 참조를 만들어뒀고, `animate()`의 애니메이터가 매 프레임 `neonRef.mat.opacity = base * hoverBoost`로 반영한다. `onMouseMove`가 `userData.isContent` 히트 시 `setHoveredFort()`로 부스트를 1.6배 올리고 `#hover-tip`에 제목 + "더블클릭으로 보기"를 띄운다. 터치 기기에서는 숨김.
- ✅ **RESET CORE 버튼 아이콘화** — 텍스트 버튼을 42px 원형 글래스 버튼(`.hud-button`)으로 교체, `stroke="currentColor"` 인라인 SVG(회전 화살표), `aria-label`/`title`/`data-tooltip`="시점 초기화". 데스크톱(`hover: hover` and `pointer: fine`)에서 `::after` 툴팁. 텍스트 버튼 시절의 padding 오버라이드 미디어쿼리는 없앴다(있었다면 원형이 찌그러졌을 것).
- ✅ **모바일 대응 세트** — `getViewportSize`/`getResponsiveFOVBoost`/`getResponsiveCamOffset`을 이식해 세로 화면에서 FOV를 부스트하고 카메라 오프셋을 더 뒤로 뺀다(MIN_FOV 15/MAX_FOV 100 유지). 두 손가락 핀치 줌(`onTouchStartPinch`/`onTouchMovePinch`/`onTouchEndPinch`). `#zoom-controls` +/− 버튼이 `body.coarse-pointer`일 때만 보이고, 탭 1회 8만큼 이동 + 120ms 반복, `stopPropagation()`으로 바다 클릭 내비게이션과 분리. 터치 안내 문구("🚢 탭해서 항해" 등).
- ✅ **더블클릭 판정 정식화** — 수동 `lastClickTime` 타이밍 판정을 제거하고 `pointerdown`(단일: 노드/바다)과 `dblclick` 이벤트(빌보드 포커스)로 분리했다. `dblclick`은 모바일 Chrome/Safari에서 더블탭 시 네이티브로 발생하므로 터치 호환도 유지된다.

### 구현 이력 / 겪은 이슈
- **줌 버튼-안내 pill 겹침(수정 완료)**: 390×844 세로 화면에서 `#zoom-controls`의 `bottom: 7rem`(city 값 그대로 이식)이 이 페이지의 `.guide-box`(한 줄로 오는 짧은 문구라 city의 3줄 스택형 세로 pill보다 얕음) 상단과 실제로 겹치는 것을 `preview_eval`의 bounding-rect 겹침 판정으로 발견했다 — `zoom.bottom(732) > guide.top(727.9)`로 4px 겹침. `bottom: 9.5rem`으로 올려 해결하고 겹침 없음을 재확인했다.
- water는 `window.onload = boot()`로 부팅해 city의 top-level `init(); animate();` 즉시 호출 패턴보다 TDZ 위험이 낮지만, 그래도 `ZOOM_BUTTON_STEP` 등 상수는 `MIN_FOV`/`MAX_FOV` 옆 최상단에 그대로 배치했다(city에서 이 실수로 로딩이 영구히 멈춘 사례가 두 번 있었음 — city/README.md 참고).
- 새로 만든 `pointerdown`/`dblclick` 리스너는 `window`가 아니라 `renderer.domElement`(canvas)에 붙였다(city와 동일) — HUD/노드트래커 등 UI 위에서의 클릭이 3D 레이캐스트로 새지 않도록.

### C. 오션 월드 그래픽 개선 (완료)

현재 씬의 문제였던 것: 수면이 조명에 씻겨 평평한 청록 단색으로 보이고, 빌보드 4개가 x=-25 한쪽에 일렬이라 "복도" 느낌이며, 근경 씨너리가 전무해 광활함이 곧 허전함으로 읽혔다. C1~C6 전 항목 완료.

- ✅ **C1. 밤바다 분위기 재조정** — `AmbientLight` 1.15 → 0.62, 바다 재질(`createOcean()`의 `seaColorMat`) 색상 `0x08314a` → `0x051f30`로 낮춰 정합. 어두워진 만큼 요트 트림(`trimMat.emissiveIntensity` 0.45→0.62)과 요새 베이스(`baseMat`에 emissive 추가)로 가독성 보정. `createGlitterPath()`가 달빛 글리터 경로 담당 — `pivot.rotation.y = atan2(-moonDirX, -moonDirZ)`로 달 방위(-900,-2100)를 향해 정렬한 `PlaneGeometry(90, 2600)`.
  - **밴드 스크롤 → 그라디언트+스파클 전면 교체(수정 완료)**: 원래 구현은 `CanvasTexture`에 수평 시머 밴드를 그려 넣고(`repeat.y=22`) 그 텍스처의 y-offset을 스크롤시키는 방식이었다. 슬로우다운(10배)까지 거쳤지만 사용자가 스크린샷에서 "줄무늬 컨베이어 벨트"처럼 보이고 "뭔가 흘러가는" 느낌이 거슬린다고 재차 지적 — 어떤 속도로 스크롤하든 밴드 구조 자체가 규칙적인 줄무늬로 읽히고, 텍스처 오프셋 애니메이션 자체가 방향성 있는 흐름을 암시하는 게 근본 문제였다. `createGlitterTextureCanvas()`(밴드)와 텍스처 스크롤 애니메이터를 완전히 삭제하고 두 요소로 교체했다: (a) `createGlitterGradientTexture()` — 내부 밴딩이 전혀 없는 단일 그라디언트(중심선만 밝고 가로/세로 모두 페더링되어 투명해지는 `CanvasTexture`)를 오퍼시티 0.2~0.3 대역에서 전체 레인 단위로만 숨쉬듯 흔들고(`mat.opacity`), 텍스처 오프셋은 아예 건드리지 않는다. (b) `createGlitterSparkles()` — 레인 영역(레인 로컬 좌표 기준 z -80~-1260, x ±17) 안에 흩뿌린 120개짜리 `Points` 가산 필드. 각 포인트는 생성 시점에 위치가 고정되고, 매 프레임 자신만의 위상(phase)·속도(speed)로 밝기(vertex color)만 재계산해 덮어쓴다 — 위치는 절대 다시 쓰지 않으므로 "제자리에서 반짝임"만 있고 이동/흐름은 전혀 없다. `preview_eval`로 1.5초 간격 스크린샷 두 장을 비교해 모든 요소(허트/등대/빌보드/스파클)가 픽셀 단위로 동일한 위치에 있고 줄무늬가 어디에도 없음을 확인했다.
- ✅ **C2. 항로 재배치 (이후 되돌림 — 아래 참고)** — `WORLD_DATA.forts`가 x=-25/+25/-25/+25로 교차 배치되고, `createFortressBillboards()`에서 `group.rotation.y = f.pos[0] < 0 ? Math.PI/2 : -Math.PI/2`로 스크린이 항상 항로(x=0) 쪽을 보게 일반화했다. `focusOnBillboard()`를 하드코딩(`worldPos.x+22`, `rotation.y=Math.PI/2`)에서 **`obj.getWorldDirection()`으로 얻은 스크린 월드 법선 벡터 기반 계산**으로 일반화 — `camTarget = worldPos + normal*22`, 회전은 스크래치 `Object3D`(`_focusTempObj`)를 그 위치에 놓고 `lookAt(worldPos)`한 뒤 그 `.rotation`을 트윈 타깃으로 사용. 좌/우 양쪽 모두 `preview_eval`로 카메라가 스크린의 "앞쪽"(법선 방향)에 착지함을 좌표로 확인(-25 쪽: 카메라 x=-2.45 vs 스크린 x=-24.45; +25 쪽: 카메라 x=2.45 vs 스크린 x=24.45).
  - **되돌림 (수정 완료)**: x=-25/+25 교차 배치가 실사용에서 체험을 망가뜨렸다 — 카메라가 `CAM_OFFSET(18,12,25)`로 항상 요트의 +x/+z 쪽에 있고 -x 방향을 바라보는 구조라, +x 쪽 요새의 스크린은 항로(-x 방향)를 향해 뒤돌아 서게 되어 카메라에서는 스크린의 "뒷면"만 보였다. `PlaneGeometry`+`MeshBasicMaterial`은 기본이 `FrontSide`라 뒷면은 레이캐스트가 아예 히트하지 않는다 — 결과적으로 4개 중 2개(+x 쪽) 빌보드는 클릭/더블클릭이 완전히 불가능했다. 전부 x=-25 한쪽으로 되돌리고(z는 15/-10/-35/-60 유지) `group.rotation.y = Math.PI/2` 고정으로 단순화했다. C2에서 일반화한 `focusOnBillboard()`(법선 벡터 기반 카메라 배치)는 한쪽 배치에도 그대로 잘 동작하고 하드코딩보다 나으므로 유지했다 — `preview_eval`로 4개 전부 레이캐스트 히트 + 더블클릭 → 올바른 제목의 모달 오픈 → Escape로 정상 종료를 확인했다. 비워진 +x 쪽은 C7(해상 마을)로 채웠다.
- ✅ **C3. 근경 씨너리** — `createLighthouse()`(색 밴드 3단 + 하우징 + 회전 빔 2개(가산 콘, `beamPivot.rotation.y = time*0.0006`) + 에미시브 전구, 실제 라이트 없음), `createIsletsAndWreck()`(암초 섬 — `IcosahedronGeometry(1,0)` 랜덤 스케일/회전 클러스터 + 가산 폼 링, 반쯤 잠긴 난파선 1척 — roll 28°/yaw 22° 기운 헐 + 부러진 마스트).
  - **등대 재배치(수정 완료)**: 원래 x=45,z=-22(+x 쪽)에 있었는데, C7 재배치와 같은 이유(카메라 시야 원뿔이 -x/-z 쪽을 향함)로 회전 빔이 사실상 보이지 않았다. `(-85, -90)`로 옮겨 -x 배경 마을 뒤쪽에서 실제로 스윕하는 빔이 보이도록 했다 — `preview_eval`로 n0~n3 노드 기준 NDC x가 화면 안(-0.86~0.06)에 들어오는 것을 확인했다(n4에서는 -1.4로 화면 밖이지만, 배경 요소라 모든 노드에서 보일 필요는 없다고 판단).

- ✅ **C6. 원경 환초 점등** — `createScrapAtolls()`의 각 타워에 1~2개의 작은 경고등 점(공유 `atollWarnGeo`/`atollWarnMatWarm`/`atollWarnMatRed`)을 추가, 절반 정도는 빨간색으로 사각파(square-wave) opacity 블링크(랜덤 위상).
- ✅ **C7. 해상 마을 (신규 → -x 배경으로 재배치 → 스케일 확대, 완료)** — 요트 항로를 따라 항해할 때 마을이 "항상 눈에 들어오는" 존재감을 갖도록 배치를 다시 잡았다. `createVillage()`가 `VILLAGE_CLUSTERS`(4개 클러스터, x=-48~-62, z=-8/-35/-65/-100) 각각에 `createStiltHut()`으로 만든 소형 수상 가옥 3~4채(박스 몸체 + 피라미드 지붕 + 다리 4개 + 창문 1~2개, 공유 지오메트리/머티리얼에 스케일·회전만 변주)를 반원(레인에서 먼 -x 쪽) 배치한다. 클러스터 내 인접 가옥끼리 얇은 데크+포스트 부잔교(`pierGroup`)로 연결하고, 클러스터당 1~2척의 정박 소형 보트(박스 헐)를 부잔교 근처에 배치해 천천히 흔들리게 했다(`animators`에서 `Math.sin(time*0.0004~0.0006)`). 두 곳에 "마을 중심" 랜턴 포컬 포인트(`createLanternFocalPoint(x,z)` 헬퍼, z=-35/-100 근처)를 두어 — 다리 달린 원형 플랫폼 위에 랜턴 기둥과 에미시브 스피어 4개 + `createBuoyHaloTexture()` 재사용 가산 헤일로 스프라이트. 암초 섬 2개(`createIsletsAndWreck()`의 `islets` 배열, x=-70/z=-22와 x=-68/z=-50)를 클러스터 사이에 끼워 넣어 "암초 위에 지은 마을" 느낌을 보강했다. 모두 `interactiveObjects`에 넣지 않았고(비인터랙티브), 새 `PointLight` 없이 에미시브 메시 + 가산 스프라이트로만 발광감을 냈다.
  - **최초 배치(+x, 카메라 시야 밖) → 좌표 재계산 → -x 배경 재배치(수정 완료)**: 첫 버전은 (되돌리기 전) x=-25/+25 교차 배치 시절에 비게 된 +x 쪽(x=16~22)에 마을을 뒀는데, 사용자가 스크린샷에서 "부잔교 모서리가 화면 우측 끝에 겨우 걸리는 정도"라고 재차 지적했다 — 근본 원인은 `CAM_OFFSET(18,12,25)`가 요트의 +x/+z 쪽에 카메라를 놓고 `lookAt`이 요트+(-12,1.5,-18) 방향, 즉 -x/-z 쪽을 향하게 하는 구조라는 데 있다: +x 쪽 씬너리는 카메라와 "같은 쪽"에 있으므로 화면 가장자리 이상으로는 절대 들어올 수 없고, -x 쪽(빌보드가 있는 쪽)은 시야 원뿔 한가운데에 있다. 그래서 마을 전체를 -x 배경(x=-45~-70, 빌보드의 x=-25보다 더 바깥쪽)으로 옮겼다.
  - **z 스태거를 감(感)이 아니라 계산으로 확정**: 단순히 "노드마다 하나씩 앞에 오도록"이 아니라, `animate()`가 실제로 쓰는 것과 동일한 `camera.position`/`camera.lookAt` 계산을 `preview_eval`로 재현한 뒤 후보 좌표를 `Vector3.project(camera)`로 NDC에 투영해 화면 안(|x|≲1)에 들어오는지 직접 측정했다. 그 결과 x=-45~-65 대역에서는 "요트 현재 z보다 약 40~90 유닛 더 앞(-z 방향)"에 있어야 NDC x가 화면 안(대략 -1.0~0.4)에 들어온다는 규칙을 확인했고, 이를 근거로 클러스터 z를 -8/-35/-65/-100으로 확정했다 — 5개 노드(z=40/15/-10/-35/-60) 전부에서 최소 2개 클러스터가 화면 안에 들어오는 것을 노드별 NDC 표로 남겨뒀다(코드 주석 참고). 가장 빡빡했던 n4(z=-60)가 마지막 클러스터를 z=-70이 아니라 z=-100까지 밀어낸 이유다(z=-70이었을 때 NDC x=-1.5~-1.7로 화면 밖이었다).
  - **검증**: n0~n4 전 노드에서 `preview_screenshot`으로 마을(허트/창문/랜턴/등대 중 최소 하나)이 화면에 걸리는 것을 확인했고, 4개 빌보드 전부 `dblclick` → 올바른 제목의 모달 오픈 → `Escape` 닫힘을 재확인해 마을이 레이캐스트를 가리지 않음을 검증했다.
  - **재배치 후에도 여전히 너무 작다는 재지적 → 클러스터 단위 스케일 배율로 확대(수정 완료)**: -x 배경으로 옮긴 뒤에도 사용자가 "마을이 여전히 거의 안 보인다"고 재차 지적했다 — 30~55 유닛 거리에 스케일 1.6~2.4짜리 오두막은 이 씬의 광각(FOV 40, 안개 시작 120)에서 지붕 실루엣 몇 픽셀로만 읽혀 "특징 없는 점"에 가까웠다. 개별 지오메트리(허트 몸체/지붕/창문/부잔교/보트)를 하나하나 키우는 대신, `clusterGroup.scale.setScalar(VILLAGE_CLUSTER_SCALE)`로 클러스터 전체(허트+부잔교+보트)를 한 번에 균등 확대하는 쪽을 택했다 — 개별 메시 스케일을 다 따로 조정하는 것보다 코드 변경이 훨씬 적고, 클러스터 내부 상대 비율(허트-부잔교 길이, 부잔교-보트 위치)이 저절로 유지된다. `createStiltHut()`의 `waterlineY`는 허트 로컬 원점 기준(스케일 이전) 정의라 `clusterGroup.scale`이 로컬 프레임 전체(y=0 수면 포함)를 균등하게 늘리므로, 다리(leg)가 여전히 (확대된) 수면선에서 끝나는 것이 그대로 성립한다 — 별도 waterline 보정 없이도 다리가 물에 붕 뜨거나 파고드는 문제가 생기지 않음을 확인했다.
    - 1차 시도로 `VILLAGE_CLUSTER_SCALE = 2.0`(허트 로컬 스케일은 그대로 1.6~2.4)을 넣어봤는데, `preview_screenshot`으로 확인해보니 오두막이 빌보드 스크린보다도 훨씬 크게 화면을 압도해 "배경 마을"이 아니라 "전경 잡동사니"처럼 보였다(빌보드 레이캐스트를 실제로 가리지는 않았지만, 시각적 존재감이 과했다). 최종적으로 `VILLAGE_CLUSTER_SCALE = 1.4` + 허트 로컬 스케일 범위를 2.1~3.0으로 재조정(유효 스케일 ~3.0~4.2)해 "지붕+창문이 한눈에 보이는 뚜렷한 건물"과 "빌보드를 압도하지 않는 배경 스케일" 사이의 균형점을 잡았다.
    - 창문(`hutWindowMat`) 색을 `0xffcf8a` → `0xffe0a0`(더 밝고 따뜻하게)로, `PlaneGeometry` 크기를 0.6→1.0으로 키워 야간 시인성을 높였다. 랜턴 포컬 포인트의 헤일로 스프라이트도 9→13으로 키우고(`centerGroup`도 동일한 `VILLAGE_CLUSTER_SCALE`을 적용해 랜턴 기둥/구체가 허트와 비례를 맞추도록 함), 등대(`createLighthouse()`)에도 `LIGHTHOUSE_SCALE = 1.5`를 그룹 전체에 적용해 커진 전경 마을에 맞춰 스카이라인 앵커도 함께 키웠다.
    - **오클루전 재검증**: 마을/등대 확대·재배치가 빌보드 레이캐스트를 가리지 않는지 `preview_eval`로 직접 확인했다 — 5개 노드 각각에서 `animate()`와 동일한 카메라 위치/lookAt을 재현한 뒤, 카메라에서 4개 빌보드 스크린 각각으로 레이를 쏴 `raycaster.intersectObjects(scene.children, true)`의 첫 히트가 항상 스크린 자기 자신인지 확인했다(20개 조합 = 5노드 × 4빌보드 전부 통과, 첫 히트 거리가 스크린까지의 실제 거리와 일치). r128에서는 `Raycaster`에 `camera`를 명시적으로 설정하지 않으면 `Sprite.raycast()`가 `Cannot read properties of null (reading 'matrixWorld')`로 예외를 던진다는 것도 이 과정에서 발견했다(부표/랜턴 헤일로 스프라이트가 씬에 있어서) — 실제 앱 코드 경로에는 영향 없는 테스트 스크립트 이슈였지만, `raycaster.camera = camera`(또는 임시 카메라)를 설정해야 한다는 점을 기록해둔다.

### D. 빌보드 포커스 카메라 회전 — Euler 컴포넌트 트윈 → 쿼터니언 slerp (완료)

- ✅ **카메라가 엉뚱한(먼) 방향으로 도는 문제(수정 완료)**: 빌보드를 더블클릭했을 때 카메라가 화면 밖으로 크게 돌았다가 다시 스크린으로 돌아오는 등 "먼 길로" 회전한다는 사용자 보고가 있었다. 근본 원인은 `focusOnBillboard()`/`closeModal()`이 카메라 회전을 `TWEEN.Tween(camera.rotation).to({x,y,z}, ...)`로, 즉 Euler(x/y/z) 각도 3개를 각각 독립적으로 선형 보간하는 방식이었다는 데 있다 — 이 방식은 서로 다른 두 가지 이유로 항상 깨진다:
  1. **±2π 랩어라운드**: 예를 들어 현재 `y=-3.0 rad`, 목표 `y=+1.6 rad`이면 컴포넌트 lerp는 그냥 -3.0에서 +1.6까지 4.6 rad을 그대로 쭉 훑는다. 실제로는 -3.0과 +1.6이 원 위에서 반대 방향으로 약 1.68 rad만 가면 되는 관계인데, Euler 숫자 자체는 "이게 원 위의 각도"라는 걸 모르기 때문에 항상 둘 중 더 긴 경로를 훑을 위험이 절반이다.
  2. **짐벌 동치 표현**: `lookAt()`이 반환하는 Euler는 예를 들어 `(x≈π, y', z≈π)`처럼, 실제로는 어떤 `(x≈0, y'', z≈0)`과 완전히 동일한 방향(orientation)을 나타내지만 x/y/z 숫자 자체는 전혀 다른 경우가 있다. 두 Euler가 "비슷한 방향"을 서로 다른 숫자 조합으로 표현하고 있으면, 컴포넌트별 lerp는 시작도 끝도 실제로 거친 적 없는 중간 방향들을 오가며 요동친다.
  - **해결**: 회전 트윈을 쿼터니언 slerp(구면 선형 보간)로 교체했다. 쿼터니언은 회전을 원(circle)이 아니라 회전 매니폴드(4차원 단위 구 위의 점)로 표현하므로, slerp는 두 쿼터니언 사이의 항상 최단 호를 따라 보간한다 — Euler 숫자가 어떻게 나왔는지와 무관하게 위 두 실패 모드가 구조적으로 발생하지 않는다.
  - **구현**: `focusOnBillboard()`는 그대로 스크래치 `_focusTempObj.lookAt(worldPos)`로 목표 방향을 계산하되, `.rotation`(Euler) 대신 `.quaternion`을 트윈 타깃으로 읽는다. 실제 트윈은 `camera.rotation`이 아니라 `{t:0}→{t:1}` 진행률 객체를 같은 duration/easing으로 트윈하면서 `onUpdate`에서 `camera.quaternion.slerpQuaternions(startQuat, endQuat, o.t)`를 호출하는 방식으로 바꿨다 — r128 소스(`three.min.js`)를 직접 확인해 `Quaternion.prototype.slerpQuaternions(qa, qb, t)`가 r128에 이미 존재함을(`slerpQuaternions(t,e,n){this.copy(t).slerp(e,n)}`, 내부적으로 `copy().slerp()`와 동일) 확인하고 그대로 사용했다. `onComplete`에서 `camera.quaternion.copy(endQuat)`으로 부동소수 오차를 정리한 뒤 모달을 연다. `closeModal()`의 복귀 트윈도 동일한 패턴으로 바꿨고, 카메라 위치 트윈(`camera.position`)은 그대로 뒀다 — 이 버그는 회전에만 있었다.
  - **원상태 저장 갱신**: `originalCamState`에 `quat: new THREE.Quaternion()` 필드를 추가해 `focusOnBillboard()` 진입 시 `camera.quaternion`을 스냅샷하고, `closeModal()`이 그 쿼터니언으로 slerp 복귀하도록 했다. 기존 `rot`(Euler) 필드는 코드 스타일상 남겨뒀지만 더 이상 트윈에 쓰이지 않는다.
  - **`animate()` 카메라 팔로우와의 상호작용 재확인**: 매 프레임 카메라를 요트에 붙이는 `lookAt` 팔로우 로직은 이미 `!isInspecting`으로 가드돼 있고, `focusOnBillboard()`가 트윈 시작 "전에" `isInspecting = true`를 설정하는 순서도 그대로였다 — 회전 트윈 방식만 바꿨을 뿐 이 가드/순서에는 손대지 않았다.
  - **검증**: `preview_eval`로 `focusOnBillboard()`를 직접 호출하면서 매 `requestAnimationFrame`마다 `camera.quaternion.angleTo(endQuat)`을 샘플링해, 값이 시작값(예: n4에서 z=15 지점 빌보드로 포커스 시 약 2.12 rad ≈ 121°)에서 0까지 프레임마다 단조 감소하고 중간에 한 번도 증가하지 않음을 확인했다(0회 증가 관측). 여러 노드(n0~n4)에서 여러 빌보드로 개별 포커스를 걸어 같은 방식으로 재확인했고, 실제 UI 경로(캔버스에 `dblclick` 이벤트 디스패치 → 모달 오픈 → `Escape`로 닫힘 → `isInspecting` false로 복귀)도 별도로 확인했다.
  - **테스트 스크립트 부산물**: 검증 스크립트에서 `raycaster.intersectObjects(scene.children, true)`를 직접 호출했을 때 r128의 `Sprite.raycast()`가 `raycaster.camera`가 설정되지 않으면 `Cannot read properties of null (reading 'matrixWorld')`로 예외를 던지는 것을 발견했다(부표/랜턴 헤일로 스프라이트 때문) — 실제 앱의 클릭/더블클릭 핸들러는 영향받지 않지만(그쪽은 다른 경로), 씬에 스프라이트가 있는 상태에서 임의 카메라로 레이캐스트 디버깅 스크립트를 짤 때는 `raycaster.camera = camera`를 먼저 설정해야 한다는 점을 기록해둔다.

### 구현 이력 / 겪은 이슈 (C1-C6)
- **글리터 경로가 안 보이던 문제(수정 완료)**: 처음 만든 달빛 글리터 경로(`createGlitterPath()`)가 좌표/텍스처/오퍼시티 모두 정상인데도 기본 카메라 뷰에서 거의 안 보였다. `preview_eval`로 바다 평면(`outerSea`)을 잠깐 `visible=false`로 꺼보니 글리터 경로가 선명하게 나타나는 것을 확인 — 대형 정적 바다 평면(`depthWrite` 기본 true)이 같은 y≈0 높이의 가산 블렌딩 평면을 깊이 테스트로 가려버린 것이었다. `lane.renderOrder = 5`(바다 평면들의 기본 0보다 높게)로 해결. 가산 블렌딩 + `depthWrite:false` 조합이어도 depthTest 자체는 여전히 기본 활성이라는 점을 놓치기 쉬움 — 비슷한 오버레이 평면을 추가할 때 다시 겪을 수 있는 패턴이라 기록.
- **등대 빔이 하늘을 뒤덮던 문제(수정 완료)**: 첫 시도에서 빔 콘 크기를 `ConeGeometry(9, 90, ...)`로 잡았는데, 등대가 요트 항로에서 그리 멀지 않은 x=45/z=-22에 있다 보니 근경 카메라 기준으로 화면 대부분을 뒤덮는 거대한 회색 삼각형으로 렌더링됐다. 요트 스케일(~9유닛)과 등대 자체 타워 높이(~20유닛) 기준으로 `ConeGeometry(2.2, 26, ...)`까지 축소해 자연스러운 크기로 맞췄다.
- **글리터 경로 방위각 계산**: 로컬 -z축이 달 방향을 향하도록 피벗을 Y축 회전시킬 때, THREE의 Y-회전 행렬에서 로컬 (0,0,-1)이 월드 (-sinθ, 0, -cosθ)로 매핑되는 것을 유도해 `theta = atan2(-moonDirX, -moonDirZ)`로 계산했다. 처음에 단순히 `atan2(moonDirX, moonDirZ)`를 썼다가 반대 방향(카메라 뒤쪽)에 렌즈 형태로 나타나는 것을 `preview_eval`의 월드 좌표 확인으로 발견하고 수정했다.
- **`focusOnBillboard()` 일반화**: `obj.getWorldDirection()`으로 스크린의 로컬 +z(법선)를 월드 공간으로 얻고, `worldPos + normal*22`를 카메라 목표 위치로 삼았다. 회전은 스크래치 `THREE.Object3D`(`_focusTempObj`, 모듈 스코프에 한 번만 생성)를 그 목표 위치에 놓고 `lookAt(worldPos)`한 뒤 결과 `.rotation`을 트윈 타깃으로 읽는 방식 — 기존 코드 스타일(임시 오브젝트로 회전 계산 후 트윈)과 일관되게 맞췄다. 좌우 양쪽 다 `preview_eval`로 카메라와 스크린의 x좌표 관계를 직접 찍어 "정면에 착지"함을 검증했다(단순히 스크린샷만으로는 앞/뒤 구분이 애매할 수 있어서 좌표 비교가 더 확실했다).
- **애니메이터 시그니처**: `animators[]`의 각 함수는 `fn(time)` 한 인자만 받는다(`animate()`에서 `animators.forEach(fn => fn(time))`). 등대 빔 회전을 델타타임 누적으로 짜려다 dt가 없다는 걸 뒤늦게 발견 — `beamPivot.rotation.y = time * 0.0006` 같은 절대각 방식으로 바꿔 프레임레이트에 무관하게 만들었다.
- **디스폴즈 검증**: `disposeObject3D()`를 city에서 그대로 이식하고, `renderer.info.memory.geometries` 카운트가 5회 스폰/디스폰 사이클 전후로 동일함을 `preview_eval`로 확인했다(179 → 179).
- **애니메이션 진정 패스(수정 완료)**: 사용자가 "뭔가 빠르게 흘러가는 것"이 거슬린다고 보고해, 매 프레임 애니메이터를 전수 점검했다. 발견한 순서대로:
  1. `createGlitterPath()`의 달빛 글리터 텍스처 y-offset 스크롤 — `time * 0.00012`가 방향성 있는 흐름으로 읽혀 "빠르게 흐르는 물"처럼 보임. **10배 감속**(`0.00012` → `0.000012`)하고 오퍼시티 반짝임(twinkle) 비중을 그대로 살려 흐름감보다 반짝임으로 보이게 했다.
  2. C5 폼 스페클 — 오퍼시티 변조가 `time * 0.0017`로 다소 빨랐던 것을 **약 2배 감속**(`0.0017` → `0.0008`).
  3. 갈매기(`createBackgroundLife()`) — 선회 각속도 `0.00025~0.0004`와 날갯짓 `time*0.012`를 각각 **절반으로 감속**(`0.000125~0.0002`, `time*0.006`).
  4. **가장 큰 원흉**: 부표 전구(`createMarkerBuoys()`) 오퍼시티 펄스가 `0.3 + Math.sin(time*0.01)*0.7`였다 — `time*0.01`은 60fps에서 약 1.6Hz(0.6초마다 풀스케일 왕복)의 사실상 스트로브였고, 진폭도 0.3~1.0으로 커서 "은은한 부표 불빛"이 아니라 빠르게 깜빡이는 경고등처럼 보였다. **약 3배 감속 + 진폭 축소**(`time*0.003`, `0.55 ± 0.35`)로 교체, 같은 버그가 있던 헤일로 스프라이트 오퍼시티(`time*0.01`)도 동일하게 `time*0.003`으로 맞췄다.
  5. 화물선(`updateShipTraffic`) 속도(8~14 유닛/초, 원경 x=±1400~1500)는 원래도 느려 그대로 뒀다.
  - 등대 회전빔(`time*0.0006`)과 요트 idle 부유(`time*0.0007~0.0012`)는 점검 결과 이미 충분히 느려 손대지 않았다.
  - 검증: `preview_screenshot`을 1초 간격으로 두 장 찍어 글리터 레인의 시각적 위치가 눈에 띄게 "점프"하지 않음을 확인했고, 변경된 각 계수를 `preview_eval`로 소스에서 재확인했다.

### 구현 이력 / 겪은 이슈 (글리터 재작업 + 마을 재배치)
- **밴드 텍스처는 감속으로 못 고친다는 결론**: 텍스처 스크롤 감속(위 항목)까지 거쳤는데도 "줄무늬"라는 지적이 다시 나온 것을 보고, 문제를 "속도"가 아니라 "구조" 문제로 재정의했다 — 밴드 자체가 규칙적인 반복 패턴인 이상 스크롤을 아무리 늦춰도 결국 사람 눈에는 "정렬된 줄이 지나간다"로 읽힌다. 그래서 애니메이션을 손보는 대신 밴드 텍스처를 완전히 없애고 무-패턴 그라디언트(중심선 페더링만 있고 반복 요소가 전혀 없는 텍스처)로 바꾸는 쪽을 택했다. 반짝임은 텍스처가 아니라 개별 스파클 포인트가 담당하도록 역할을 분리한 것도 같은 이유 — 하나의 텍스처가 "은은한 빛"과 "반짝임"을 동시에 표현하려다 보니 어느 한쪽으로도 깔끔하지 않았다.
- **스파클을 "포인트별 독립 위상"으로 구현**: `PointsMaterial`의 전체 `opacity`만 흔들면 모든 점이 같은 리듬으로 깜빡여 여전히 하나의 덩어리 패턴처럼 보일 위험이 있었다. 대신 각 포인트에 고유한 `phase`/`speed`를 부여하고, 매 프레임 `color` 버퍼 어트리뷰트(`colorAttr.array`)를 순회하며 `baseColor * brightness(time, phase, speed)`로 덮어써 `needsUpdate = true`로 반영했다. 포인트 120개 × RGB 3개 = 360 float 갱신은 프레임당 비용이 무시할 만한 수준이라 별도 최적화 없이 그대로 뒀다. 위치(`position`) 어트리뷰트는 생성 시 한 번만 쓰고 이후 절대 건드리지 않아 "반짝이지만 움직이지 않는다"를 구조적으로 보장했다.
- **화면 내 배치를 감(感)이 아니라 `camera.project()`로 검증하는 패턴 확립**: 이번 마을 재배치에서 처음 확립한 방법 — 스크린샷을 보고 "왼쪽에 있으니 보이겠지"라고 추측하는 대신, `preview_eval`에서 `animate()`와 동일한 카메라 위치/lookAt 계산을 재현한 `camera.clone()`을 만들어 두고, 후보 월드 좌표를 `Vector3.project(camera)`로 NDC에 투영해 `|x| < 1`인지 직접 읽었다. 이 방법으로 "노드마다 후보 클러스터 하나씩 재보기"를 반복문으로 한 번에 처리할 수 있었고, 실제로 마지막 마을 클러스터 z를 -70에서 -100까지 밀어야 했던 것도 이 계산에서 나왔다(스크린샷만으로는 "어느 정도 멀어야 화면 안에 들어오는지"를 정량화하기 어려웠다). 카메라 시야에 걸리는지가 관건인 배경 씬너리를 추가/재배치할 때 재사용할 만한 패턴이라 기록해둔다.
